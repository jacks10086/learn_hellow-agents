> 本章记录 chain.py 模块的设计方案

## 一、chain.py 模块架构

### 1.1 模块职责

```
┌─────────────────────────────────────────┐
│           chain.py 核心职责              │
├─────────────────────────────────────────┤
│  将多个工具串联执行，前一步输出作为后一步输入  │
└─────────────────────────────────────────┘
```

**实际场景**：用户问"北京今天天气如何？需要带伞吗？"

```
步骤1: search("北京今天天气") → "北京晴，25度"
步骤2: analyze("北京晴，25度，需要带伞吗？") → "不需要带伞"
```

### 1.2 类设计

```
┌──────────────────┐      ┌──────────────────┐
│    ToolChain     │      │ ToolChainManager │
├──────────────────┤      ├──────────────────┤
│ - name           │      │ - registry       │
│ - description    │      │ - chains: Dict   │
│ - steps: List    │      ├──────────────────┤
├──────────────────┤      │ + register_chain()│
│ + add_step()     │      │ + execute_chain() │
│ + execute()      │      │ + list_chains()   │
└──────────────────┘      │ + get_chain_info()│
                          └──────────────────┘
```

### 1.3 核心数据结构

```python
# 步骤定义
step = {
    "tool_name": "search",           # 调用哪个工具
    "input_template": "{input}",     # 输入模板
    "output_key": "search_result"    # 结果存到哪个变量
}

# 执行上下文
context = {
    "input": "北京天气",              # 初始输入
    "search_result": "晴，25度",      # 步骤1的输出
    "calc_result": "30"              # 步骤2的输出
}
```

### 1.4 执行流程

```
execute() 方法流程：

┌─────────────────┐
│  初始化 context  │  context = {"input": 原始输入}
└────────┬────────┘
         ▼
┌─────────────────┐
│   遍历 steps     │  for i, step in enumerate(self.steps)
└────────┬────────┘
         ▼
┌─────────────────┐
│  模板变量替换    │  input_template.format(**context)
└────────┬────────┘
         ▼
┌─────────────────┐
│   执行工具       │  registry.execute_tool(tool_name, actual_input)
└────────┬────────┘
         ▼
┌─────────────────┐
│  保存结果到上下文 │  context[output_key] = result
└────────┬────────┘
         ▼
      下一步骤...
```

### 1.5 设计意图

| 设计选择 | 理由 |
|----------|------|
| 用 `Dict` 存步骤配置 | 灵活，易于序列化/反序列化 |
| 用 `input_template` 字符串 | 支持动态变量替换，解耦步骤间依赖 |
| 用 `context` 字典传递数据 | 避免全局变量，所有步骤共享状态 |
| `ToolChainManager` 分离管理 | 单一职责，工具链定义与执行管理分离 |

## 二、管理器模式

### 2.1 核心思想

```
┌─────────────────────────────────────────┐
│  把"创建/查找/协调多个对象"的职责        │
│  集中到一个管理器类中                    │
└─────────────────────────────────────────┘
```

### 2.2 三大好处

| 好处 | 说明 |
|------|------|
| **统一入口** | 调用者不需要知道对象如何创建、存在哪里 |
| **生命周期管理** | 管理器负责创建、缓存、销毁对象 |
| **解耦** | 调用者和被管理对象不直接依赖 |

### 2.3 代码示例

```python
# 没有管理器：用户自己管理
chain1 = ToolChain("search_chain", "搜索链")
chain2 = ToolChain("calc_chain", "计算链")
# 用户要自己持有这些实例... 很乱

# 有管理器：统一注册、查找、执行
manager = ToolChainManager()
manager.register_chain(chain1)
manager.register_chain(chain2)

# 通过名字使用
manager.execute_chain("search_chain", "天气")
manager.get_chain_info("calc_chain")
```

### 2.4 C++/Qt 类比

```cpp
// Qt 中大量使用管理器模式
QNetworkAccessManager network;  // 网络请求管理器
QFileManager fileManager;       // 文件管理器
QPluginManager pluginManager;   // 插件管理器

// 用法类似：通过管理器操作底层对象
QNetworkReply* reply = network.get(request);
```

| Qt 管理器 | Python 对应 |
|-----------|-------------|
| `QNetworkAccessManager` | `ToolChainManager` |
| 管理 `QNetworkReply` | 管理 `ToolChain` |
| 统一发起网络请求 | 统一执行工具链 |

### 2.5 什么时候用？

**✅ 适合的场景**：

| 场景 | 例子 |
|------|------|
| 需要管理多个同类型对象 | 多个工具链、多个连接池 |
| 对象创建复杂 | 需要读取配置、依赖注入 |
| 需要全局访问点 | 日志管理器、配置管理器 |
| 需要缓存/复用 | 数据库连接池 |

**❌ 不适合的场景**：

| 场景 | 原因 |
|------|------|
| 只有一个对象 | 直接用单例或模块级变量 |
| 对象之间无关联 | 不需要统一管理 |
| 简单的 CRUD | 过度设计 |

### 2.6 与其他模式的关系

```
管理器模式
    │
    ├── 类似工厂模式：负责创建对象
    │
    ├── 类似注册表模式：提供查找功能
    │
    └── 类似外观模式：提供统一接口
```

```python
# 管理器 = 工厂 + 注册表 + 外观
class ToolChainManager:
    # 注册表：存储对象
    def register_chain(self, chain): ...

    # 工厂：创建或获取对象（可扩展）
    def get_or_create_chain(self, name): ...

    # 外观：简化调用
    def execute_chain(self, name, input_data): ...
```

## 三、涉及的设计模式

### 3.1 chain.py 中的模式

| 模式 | 体现 |
|------|------|
| **责任链模式** | 步骤依次执行，前一步输出传给后一步 |
| **管道模式** | 数据像水流过管道：输入 → [工具1] → [工具2] → 输出 |
| **管理器模式** | ToolChainManager 统一管理多个工具链 |
| **模板方法模式** | input_template 定义数据转换模板 |

### 3.2 责任链模式详解

```python
# 每个步骤处理输入，传递输出给下一步
step1: input → search_result
step2: search_result → calc_result
step3: calc_result → final_result
```

**C++ 类比**：Qt 事件过滤器链

```cpp
// Qt 事件过滤器链
event → [filter1] → [filter2] → [handler]
```

### 3.3 管道模式详解

```python
# 数据像水流过管道
原始输入 → [工具1] → [工具2] → [工具3] → 最终输出

# 项目代码体现
context = {"input": raw_input}
for step in self.steps:
    context[output_key] = execute_tool(step)
```

## 四、模块间关系

```
┌─────────────┐     注册工具     ┌─────────────┐
│  registry   │◄────────────────│   chain     │
│  (工具注册表) │                 │  (工具链)    │
└─────────────┘     调用工具     └─────────────┘
       ▲
       │ 组合关系
       │
┌─────────────┐
│ToolChain    │
│Manager      │ ← 管理多个 ToolChain
└─────────────┘
```

**调用链**：

```python
chain_manager.execute_chain("research", input_data)
    ↓
ToolChain.execute(registry, input_data)
    ↓
registry.execute_tool("search", query)
    ↓
Tool.run(query)
```

## 五、今日总结

### 5.1 设计模式

| 主题 | 核心理解 |
|------|---------|
| 管理器模式 | 统一入口、生命周期管理、解耦 |
| 责任链模式 | 步骤依次执行，前一步输出传给后一步 |
| 管道模式 | 数据流过多个处理节点 |
| 模板方法 | 用字符串模板定义数据转换规则 |

### 5.2 C++/Qt 对照

| Python | C++ | Qt |
|--------|-----|-----|
| `ToolChainManager` | - | `QNetworkAccessManager` |
| 责任链模式 | - | 事件过滤器链 |
| 管道模式 | `std::ranges::views` | - |
